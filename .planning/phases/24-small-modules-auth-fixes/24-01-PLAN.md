  <name>Task 1: Add row limit to asset export and MIME allowlist to invoice upload</name>
  <files>includes/Modules/Assets/Admin/class-admin-pages.php</files>
  <action>
1. In `handle_assets_export()` (around line 1507), change the unbounded query:
   ```php
   $rows = $wpdb->get_results("SELECT * FROM {$table} ORDER BY id ASC", ARRAY_A);
   ```
   to:
   ```php
   $rows = $wpdb->get_results("SELECT * FROM {$table} ORDER BY id ASC LIMIT 5000", ARRAY_A);
   ```
   5000 is a safe upper bound -- no organization should have more assets than this, and it prevents memory exhaustion on malicious repeated calls.

2. In `handle_asset_save()` (around line 1319), add a MIME type allowlist to the `wp_handle_upload` call. Change:
   ```php
   $upload = wp_handle_upload(
       $_FILES['invoice_file'],
       [ 'test_form' => false ]
   );
   ```
   to:
   ```php
   $allowed_mimes = [
       'jpg|jpeg|jpe' => 'image/jpeg',
       'png'          => 'image/png',
       'gif'          => 'image/gif',
       'pdf'          => 'application/pdf',
       'doc'          => 'application/msword',
       'docx'         => 'application/vnd.openxmlformats-officedocument.wordprocessingml.document',
       'xls'          => 'application/vnd.ms-excel',
       'xlsx'         => 'application/vnd.openxmlformats-officedocument.spreadsheetml.document',
   ];
   $upload = wp_handle_upload(
       $_FILES['invoice_file'],
       [
           'test_form' => false,
           'mimes'     => $allowed_mimes,
       ]
   );
   ```
   This blocks .php, .html, .js, .svg, and any other non-document/non-image file type. The `mimes` parameter is checked by `wp_check_filetype_and_ext()` inside `wp_handle_upload()`.

Do NOT change any other logic in `handle_asset_save()` or `handle_assets_export()`. Preserve all existing behavior.
  </action>

<task type="auto">
  <name>Task 2: Fix handle_sync_dept_members to check capability before reading POST data</name>
  <files>includes/Core/Admin.php</files>
  <action>
Rewrite `handle_sync_dept_members()` (around line 5031) to check auth BEFORE touching any `$_POST` data. The current code reads `$_POST['dept_id']` inside the `!current_user_can` branch before verifying the user is a dept manager. Restructure to:

```php
public function handle_sync_dept_members(): void {
    // Check capability BEFORE reading any POST data
    if ( ! current_user_can('sfs_hr.manage') && ! current_user_can('sfs_hr.view') ) {
        wp_die( esc_html__( 'Permission denied', 'sfs-hr' ) );
    }

    $dept_id = isset($_POST['dept_id']) ? (int)$_POST['dept_id'] : 0;
    if ( $dept_id <= 0 ) {
        wp_die( esc_html__( 'Invalid department', 'sfs-hr' ) );
    }

    // Department managers need additional scope check
    if ( ! current_user_can('sfs_hr.manage') ) {
        if ( ! $this->current_user_is_dept_manager( $dept_id ) ) {
            wp_die( esc_html__( 'Permission denied', 'sfs-hr' ) );
        }
    }

    check_admin_referer('sfs_hr_sync_dept_' . $dept_id);
    // ... rest of method unchanged from line ~5040 onward
```

Key changes:
- First line: reject any user without `sfs_hr.manage` or `sfs_hr.view` immediately, before touching POST
- Then read `$dept_id` from POST (user is already authenticated as having some HR capability)
- Then narrow scope: non-manage users must also be dept manager for this specific department
- Leave everything from `check_admin_referer` onward unchanged

The `sfs_hr.view` capability is included because department managers get this dynamically via the `user_has_cap` filter, allowing them to use the sync function for their own departments.
