**1. Workforce Status shift resolution N+1 (WADM-PERF-002):**
In `Admin_Pages.php` (around line 1182-1199), find the loop that calls `resolve_shift_for_date()` per employee. Fix:
- Before the loop, collect all employee IDs
- Run a single batch query to get shift assignments for all employees for the target date: `SELECT employee_id, shift_id FROM {$emp_shifts_table} WHERE employee_id IN (...) AND effective_date <= %s ORDER BY effective_date DESC`

- Inside the loop, use the lookup map instead of calling resolve_shift_for_date()
- Also add `LIMIT 500` to the initial employee load query (WADM-PERF-001, around line 857-864) that currently loads ALL active employees with no cap. Add pagination via offset/limit or simply cap at 500 with ORDER BY.

**2. Absent Notifications N+1 (WNTF-PERF-002):**
In `Absent_Notifications.php` (around line 203), the same N+1 shift resolution pattern exists. Apply the same batch fix:
- Before the notification loop, batch-resolve shifts for all employee IDs
- Use the lookup map inside the loop
- Also fix WNTF-PERF-001 if present: remove the duplicate call to `get_absent_employees_by_department()` — ensure it is called only once per cron run, not twice

**3. Documents uploadable types N+1 (DOC-PERF-001):**
In `class-documents-service.php`, find `get_uploadable_document_types_for_employee()` that runs a separate query per document type to check if the employee already has that type uploaded. Replace with a single batch query:
- Fetch all existing document type IDs for the employee in one query: `SELECT DISTINCT document_type_id FROM {$docs_table} WHERE employee_id = %d`
- Then filter the available types array in PHP (array_diff or similar) instead of querying per type
