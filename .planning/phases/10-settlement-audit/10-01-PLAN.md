  <files>

  <action>
Read all three PHP files. Audit each file against the 4 metrics:

**1. Security (Critical focus):**
- Every `$wpdb->query()`, `$wpdb->get_results()`, `$wpdb->get_row()`, `$wpdb->get_var()` call: Is `$wpdb->prepare()` used? Flag any raw string interpolation as Critical.
- SettlementModule table installation/migration: Does it use bare `ALTER TABLE` or unprepared `SHOW TABLES` like Loans/Core did? (Prior phases found this as Critical antipattern -- LOAN-SEC-001/002, Phase 04 Critical.)
- Handler capability checks: Does every admin-post handler verify `current_user_can()` with appropriate capability (e.g., `sfs_hr.manage`) before executing? Flag missing guards as Critical (same pattern as Leave Phase 06 -- handle_approve missing up-front cap guard).
- Handler nonce validation: Are all write operations (create/update/approve/delete settlement) protected with `wp_verify_nonce()` or `check_admin_referer()`?
- Input sanitization: Are settlement amounts, employee IDs, dates sanitized via `intval()`, `sanitize_text_field()`, or `absint()` before use?

**2. Performance:**
- Does the settlement calculation make per-employee sub-queries (attendance records, salary history) for each calculation? Count query-per-calculation patterns.
- Are settlement lists loaded without pagination (unbounded `get_results()`)? Flag for companies with 100+ terminated employees.
- Are there repeated database lookups within the same request (e.g., fetching employee data multiple times)?

**3. Duplication:**
- Cross-module: Does Settlement duplicate salary component logic from Payroll? Does it recalculate basic salary when it could reference payroll data?
- Does the EOS calculation duplicate tenure calculation that exists in Leave module (LV-CALC-001 from Phase 06 found tenure boundary evaluated at Jan 1 instead of anniversary)?
- Are there repeated date/tenure helper functions that duplicate Core helpers?
- Check overlap with EmployeeExit module (Phase 19 success criteria mentions "logical overlap with Settlement module" -- document any shared EOS logic).

**4. Logical Issues (Saudi labor law focus):**
- **Tenure brackets**: Saudi labor law (Article 84-85) specifies:
  - First 5 years: half-month salary per year of service
  - After 5 years: full-month salary per year of service
  - Verify the code implements these brackets correctly. Check edge cases: exactly 5 years, partial years, fractional months.
- **Trigger type multipliers**: Saudi labor law differentiates:
  - Resignation with <2 years service: zero entitlement
  - Resignation with 2-5 years: one-third of EOS
  - Resignation with 5-10 years: two-thirds of EOS
  - Resignation with 10+ years: full EOS
  - Termination: full EOS (no reduction)
  - Contract end: full EOS
  - Verify the code handles all trigger types correctly. Flag missing cases.
- **Salary base**: EOS should use "last salary" including basic + housing allowance. Verify what components are included in the calculation base.
- **Partial year proration**: For the final partial year, is it prorated by days/365 or months/12? Either is acceptable but must be consistent.
- **Negative/zero tenure**: What happens if settlement is created for an employee with zero days of service or a future hire date?
- **Race conditions**: Can a settlement be approved while simultaneously being edited? Is there a status lock?
- **Double settlement**: Can two settlements be created for the same employee? Is there a uniqueness check?

Rate every finding: Critical / High / Medium / Low. Include file:line references and a concrete fix recommendation for each.
