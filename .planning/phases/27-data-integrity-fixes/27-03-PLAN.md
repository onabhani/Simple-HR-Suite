<task type="auto">
  <name>Task 2: Align frontend installment calculation with admin path</name>
  <files>includes/Modules/Loans/Frontend/class-my-profile-loans.php</files>
  <action>
The admin approval path (class-admin-pages.php line 2181) calculates:
```php
$installment_amount = round( $principal / $installments, 2 );
```

But the frontend submission (class-my-profile-loans.php line 706) stores:
```php
$installment_amt = round( $monthly_amount, 2 );
```

This means if a user enters principal=10000 and monthly=700, the frontend stores installment_amount=700.00 with installments_count=15 (ceil(10000/700)=15). But the admin would compute round(10000/15, 2) = 666.67. When Payroll reads installment_amount, it gets 700.00 from the frontend path but would get 666.67 from the admin path.

Fix the frontend `validate_and_insert_loan()` method (around line 706) to use the same formula as admin:

Change:
```php
$installment_amt = round( $monthly_amount, 2 );
```
To:
```php
$installment_amt = round( $principal / $installments, 2 );
```

This ensures the stored `installment_amount` is always computed from `principal / installments_count`, regardless of submission path. The user's entered monthly amount is used to derive installments_count (which it already does at line 576-578), and then the actual per-installment amount is back-calculated for consistency.

Also verify the installments calculation above (lines 576-578) is correct:
```php
$full_months  = $monthly_amount > 0 ? (int) floor( $principal / $monthly_amount ) : 0;
$last_payment = $principal - ( $full_months * $monthly_amount );
$installments = $last_payment > 0 ? $full_months + 1 : $full_months;
```
This is fine -- it computes how many months are needed based on user-entered monthly amount. The fix only changes what gets stored as installment_amount.
  </action>

  </verify>
  <done>Frontend stores installment_amount = round(principal / installments, 2), matching the admin calculation path. No more rounding divergence between submission paths.</done>
</task>
