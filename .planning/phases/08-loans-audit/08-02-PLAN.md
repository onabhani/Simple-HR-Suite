<task type="auto">
  <name>Task 1: Read and audit Loans admin pages and dashboard widget</name>
  <files>
    includes/Modules/Loans/Admin/class-admin-pages.php (3066 lines -- loan CRUD, approval/rejection/disbursement actions, installment management, payment recording, admin list views)
    includes/Modules/Loans/Admin/class-dashboard-widget.php (273 lines -- loan summary widget on WP admin dashboard)
  </files>
  <action>
Read both PHP files. Audit each file against the 4 metrics:

**1. Security (Critical focus):**

For class-admin-pages.php:
- The `handle_loan_actions` method (registered on `admin_post_sfs_hr_loan_action`): Does it verify nonce before ANY processing? Does it check capability (e.g., `sfs_hr.manage`) before financial write operations?
- For each action type (approve, reject, disburse, edit, delete, record payment): Is there an individual capability check, or does the handler rely on a single gate at the top?
- Every `$wpdb->query()`, `$wpdb->get_results()`, `$wpdb->get_row()`, `$wpdb->get_var()` call: Is `$wpdb->prepare()` used? Flag any raw string interpolation as Critical. This is a 3,066-line file -- be thorough, check EVERY query.
- `$_POST` / `$_GET` / `$_REQUEST` usage: Is every user input sanitized before use? Check for `sanitize_text_field()`, `intval()`, `absint()`, etc.
- Output escaping: Is every rendered loan amount, employee name, date, and status properly escaped with `esc_html()`, `esc_attr()`, `wp_kses()`?
- Can a non-admin user access admin pages by guessing the URL? Check for `current_user_can()` on page callbacks.

For class-dashboard-widget.php:
- Does the widget check capabilities before rendering loan data?
- Are queries in the widget prepared?
- Is output properly escaped?

**2. Performance:**
- Admin loan list: Is it paginated or does it load all loans? Check for LIMIT/OFFSET in list queries.
- Installment schedule queries: When viewing a single loan's installments, are all installments loaded at once?
- Dashboard widget: Does it run aggregate queries (COUNT, SUM) on every admin page load? Are these cached?
- N+1 patterns: Does the loan list query trigger per-row sub-queries for employee names, installment counts, or payment status?
- Are there JOINs or subqueries that would degrade on large datasets (hundreds of loans)?

**3. Duplication:**
- Is installment calculation/generation logic in admin pages duplicated from what the frontend handler uses? Compare formulas and schedule generation code.
- Are loan status display helpers repeated between admin pages, frontend, and notifications?
- Is loan eligibility checking duplicated between admin creation and frontend request?
- Repeated formatting patterns (currency formatting, date formatting) that could be shared helpers.

**4. Logical Issues:**
- Loan approval workflow: Can a loan be approved twice? Can an already-disbursed loan be re-approved?
- Disbursement: Is the disbursement amount validated against the approved amount? Can it be changed during disbursement?
- Payment recording: Can a payment exceed the remaining balance? Can negative payments be recorded?
- Installment editing: Can installment amounts be edited after payments have been made? Does this corrupt the schedule?
- Delete: Can a loan with recorded payments be deleted? What happens to the payment records?
- Concurrent operations: If two admins approve the same loan simultaneously, is there a race condition?
- Rounding consistency: Do admin pages use the same rounding logic as the frontend for installment amounts?

Rate every finding: Critical / High / Medium / Low. Include file:line references and a concrete fix recommendation for each.
