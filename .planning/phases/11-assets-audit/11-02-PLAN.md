<objective>
Audit the Assets module's view templates -- assets-edit.php (385 lines), assets-list.php (632 lines), and assignments-list.php (447 lines) -- for security, performance, duplication, and logical issues. This plan covers ~1,464 lines of HTML/PHP rendering code.

Purpose: View templates in WordPress plugins are a common source of XSS vulnerabilities (unescaped output), inline SQL queries that bypass prepared statements, and missing CSRF tokens in forms. Prior phases found Critical unescaped output in Frontend tabs (Phase 04) and Settlement admin views that showed action buttons without capability checks (Phase 10). The assignments-list view is particularly important because it may expose asset data across employees if ownership filtering is missing.

<task type="auto">
  <name>Task 1: Read and audit Assets view templates</name>
  <files>
    includes/Modules/Assets/Views/assets-edit.php (385 lines -- asset create/edit form)
    includes/Modules/Assets/Views/assets-list.php (632 lines -- asset list table with actions)
    includes/Modules/Assets/Views/assignments-list.php (447 lines -- assignment tracking list)
  </files>
  <action>
Read all three PHP view files. Audit each file against the 4 metrics:

**1. Security (Critical focus):**
- **XSS / Output escaping**: Every dynamic value rendered in HTML must use `esc_html()`, `esc_attr()`, `esc_url()`, or `wp_kses()`. Flag any raw `echo $variable` or `<?= $variable ?>` without escaping as High/Critical depending on data source.
- **Inline SQL**: Do any views contain direct `$wpdb` queries? Views should only render data passed from the controller. Flag any `$wpdb->get_results()` or similar in view files as High (architectural violation that bypasses prepared statement patterns in the controller).
- **CSRF protection**: Does assets-edit.php include a nonce field (`wp_nonce_field()`) in the form? Flag missing nonce as Critical.
- **Capability checks in views**: Do the views check `current_user_can()` before rendering action buttons (edit, delete, assign, return)? Prior phase found Settlement admin views showing approve/reject buttons to all viewers (SADM-SEC-001 High). Check for same pattern.
- **Assignment ownership**: Does assignments-list.php filter results to show only assets assigned to the current user (for non-admin viewers)? Or does it display all assignments regardless of viewer? Flag missing ownership filter as High.
- **File upload form**: If assets-edit.php includes a file upload field, does the form have `enctype="multipart/form-data"` and is the accepted file type restricted via `accept` attribute? (Server-side validation is more important, but client-side hints are a defense-in-depth signal.)

**2. Performance:**
- Does assets-list.php render all assets without pagination, or does it have a paginated table? Flag unbounded rendering for large asset inventories.
- Are there N+1 patterns in the views (e.g., querying employee name per row inside a loop)?
- Do views make any AJAX calls or load large datasets inline?

**3. Duplication:**
- Is HTML table rendering logic duplicated across assets-list.php and assignments-list.php? Could they share a table component?
- Are status badge rendering, action button HTML, or date formatting functions repeated?
- Compare with other module view patterns (Settlement, Loans) for inconsistent approaches.

**4. Logical Issues:**
- **Form validation**: Does assets-edit.php have any client-side validation (required fields, number ranges)? Missing client-side validation is Low but worth noting.
- **Status display consistency**: Do the list views show asset status using the same status values as the admin controller? Any mismatch between displayed status and actual DB status?
- **Action availability**: Can the delete button be clicked on an assigned asset? Can the return button appear on an unassigned asset? Check for UI logic that allows invalid operations.
- **Empty states**: Do the list views handle zero-results gracefully?

Rate every finding: Critical / High / Medium / Low. Include file:line references and a concrete fix recommendation for each.
