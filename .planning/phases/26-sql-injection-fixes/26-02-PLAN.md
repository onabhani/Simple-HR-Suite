Fix the two raw LIKE clause interpolations in HiringModule.php where employee code prefix is used in a LIKE query without $wpdb->prepare().

**Line ~348-352** (in the trainee-to-employee conversion method):
```php
// BEFORE (unsafe pattern):
$last = $wpdb->get_var(
    "SELECT employee_code FROM {$wpdb->prefix}sfs_hr_employees
     WHERE employee_code LIKE '{$prefix}%'
     ORDER BY id DESC LIMIT 1"
);

// AFTER (prepared with esc_like):
$last = $wpdb->get_var(
    $wpdb->prepare(
        "SELECT employee_code FROM {$wpdb->prefix}sfs_hr_employees
         WHERE employee_code LIKE %s
         ORDER BY id DESC LIMIT 1",
        $wpdb->esc_like( $prefix ) . '%'
    )
);
```

**Line ~535-539** (in the candidate-to-employee conversion method):
Apply the exact same fix — same query pattern, same `$prefix = "USR-"` variable.

Although `$prefix` is currently hardcoded as `"USR-"` and not user-supplied, the fix is important because:
1. It establishes the correct pattern for any future changes where the prefix might be configurable
2. It eliminates a false-positive in SQL injection scanners
3. The `esc_like()` call properly escapes `%` and `_` metacharacters in the prefix value
4. It satisfies the audit requirement SQL-04 for LIKE clause fixes

Ensure the fix preserves the existing logic: find the highest employee_code starting with the prefix, extract the numeric suffix, and increment by 1.
