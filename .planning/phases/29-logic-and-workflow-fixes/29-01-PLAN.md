**LOGIC-05: Fix dynamic capability caching for AJAX requests.**

In `Capabilities::dynamic_caps()` (line 62-125), the static `$cache` array persists for the entire PHP request lifetime. This is correct for normal page loads. However, the issue manifests when:

1. A persistent object cache (Redis/Memcached) or opcache preloading causes the static to survive across what should be separate logical requests
2. More practically: when `user_has_cap` fires very early in an AJAX request before the DB is ready, the cache stores a false-negative, and all subsequent capability checks in that same AJAX request return the wrong result

The fix has two parts:

**Part A — Add request-scoping to the static cache:**

Replace the simple `static $cache = [];` with a request-scoped cache that resets if the request ID changes. Add a static request marker:

```php
static $cache = [];
static $request_id = null;

// Reset cache if this is a new request (handles persistent process edge cases)
$current_request_id = $_SERVER['REQUEST_TIME_FLOAT'] ?? microtime( true );
if ( $request_id !== $current_request_id ) {
    $cache = [];
    $request_id = $current_request_id;
}
```

This uses `REQUEST_TIME_FLOAT` which is set once per HTTP request and stays constant. If the PHP process serves multiple requests (PHP-FPM persistent workers), the float timestamp changes and the cache resets.

**Part B — Ensure the filter is registered during AJAX:**

In `ensure_roles_caps()`, the `self::$filter_added` guard prevents double-registration. However, if `ensure_roles_caps()` is never called during an AJAX request (it runs on `admin_init` for admin pages), the dynamic caps filter might not be active.

Check that the `admin_init` hook fires for AJAX (it does for `admin-ajax.php` but not for REST API calls). For REST API, the filter needs to be registered via `rest_api_init`. Add to the bottom of `ensure_roles_caps()`:

```php
// Ensure dynamic caps filter is also active during REST API requests
if ( ! self::$filter_added ) {
    add_filter( 'user_has_cap', [ __CLASS__, 'dynamic_caps' ], 10, 4 );
    self::$filter_added = true;
}
```

Wait — this code is already at lines 50-53. The issue is that `ensure_roles_caps()` must be called. Verify it's hooked to both `admin_init` and `rest_api_init`. Search `hr-suite.php` for where `ensure_roles_caps` is called and ensure it covers REST. If it only runs on `admin_init`, add `rest_api_init` hook as well.

Search `hr-suite.php` for `ensure_roles_caps` and `Capabilities::` to find where it's bootstrapped. If the call is only inside an `is_admin()` guard or only on `admin_init`, the REST API path misses it entirely. Add a `rest_api_init` callback that calls `Capabilities::ensure_roles_caps()` if not already done.

The key changes to `Capabilities.php`:
