**LOGIC-01: Fix TOCTOU races in Leave overlap check and Loan fiscal year check.**

The race condition pattern in both modules is identical:
1. Check if condition exists (overlap / fiscal year loan)
2. If no conflict, insert new row
3. Between steps 1 and 2, a concurrent request can also pass step 1

**Fix for Leave overlap (two locations):**

There are two leave request creation paths:
- Frontend shortcode handler (around line 4559 in LeaveModule.php)
- Admin handler `handle_create` (around line 6120 in LeaveModule.php)

For BOTH paths, wrap the overlap-check + insert in a transaction:

**Path 1 — Frontend shortcode handler (~line 4559):**

Before the `has_overlap()` call, add:
```php
$wpdb->query( 'START TRANSACTION' );
```

Then change `LeaveCalculationService::has_overlap()` to use a locking variant. Add a new method `has_overlap_locked()` to `LeaveCalculationService`:

```php
/**
 * Check overlap with row-level lock to prevent TOCTOU race.
 * Must be called inside a transaction.
 */
public static function has_overlap_locked(int $employee_id, string $start, string $end): bool {
    global $wpdb;
    $t = $wpdb->prefix . 'sfs_hr_leave_requests';
    $sql = "SELECT COUNT(*) FROM $t
            WHERE employee_id=%d
              AND status IN ('pending','approved')
              AND NOT (end_date < %s OR start_date > %s)
            FOR UPDATE";
    $cnt = (int)$wpdb->get_var($wpdb->prepare($sql, $employee_id, $start, $end));
    return $cnt > 0;
}
```

The `FOR UPDATE` lock prevents a second concurrent transaction from reading the same rows until the first transaction completes its insert and commits.

Replace the `has_overlap()` call with `has_overlap_locked()` in the frontend path. If overlap is detected, `ROLLBACK` and return error. After the successful `$wpdb->insert()` of the leave request (find the INSERT that creates the leave request row, typically further down where the request is inserted into `$req_t`), add `$wpdb->query('COMMIT');`.

IMPORTANT: If any error path exists between START TRANSACTION and the insert (e.g., insufficient balance, validation failure), add `$wpdb->query('ROLLBACK');` before returning the error. Trace every early-return/error path between the transaction start and the insert to ensure rollback happens.

**LOGIC-01: Fix TOCTOU race in reference number generation.**

`Helpers::generate_reference_number()` (line 837-867) reads MAX(reference_number), increments, and returns. Two concurrent calls can read the same MAX and produce the same reference number, causing a UNIQUE constraint violation on insert.

The fix: use `SELECT ... FOR UPDATE` in the MAX query so that concurrent calls block until the first transaction commits. However, `generate_reference_number` is called from various contexts — some already inside transactions (like Payroll), some not.

The safest approach is to make the function self-contained with its own transaction:

```php
public static function generate_reference_number(
    string $prefix,
    string $table,
    string $column = 'request_number',
    ?string $created_at = null
): string {
    global $wpdb;

    $year = $created_at ? date( 'Y', strtotime( $created_at ) ) : wp_date( 'Y' );
    $like_pattern = $prefix . '-' . $year . '-%';

    // Use a named lock to serialize reference number generation per prefix+year.
    // This is safer than a transaction since callers may already be inside one.
    $lock_name = 'sfs_hr_ref_' . $prefix . '_' . $year;
    $got_lock  = $wpdb->get_var( $wpdb->prepare( "SELECT GET_LOCK(%s, 5)", $lock_name ) );

    $max_ref = $wpdb->get_var(
        $wpdb->prepare(
            "SELECT MAX(`{$column}`) FROM `{$table}` WHERE `{$column}` LIKE %s",
            $like_pattern
        )
    );

    $next = 1;
    if ( $max_ref ) {
        $parts = explode( '-', $max_ref );
        $last_seq = (int) end( $parts );
        $next = $last_seq + 1;
    }

    $sequence = str_pad( $next, 4, '0', STR_PAD_LEFT );
    $result = $prefix . '-' . $year . '-' . $sequence;

    // Release the named lock
    if ( $got_lock ) {
        $wpdb->get_var( $wpdb->prepare( "SELECT RELEASE_LOCK(%s)", $lock_name ) );
    }

    return $result;
}
```

Using MySQL `GET_LOCK()` / `RELEASE_LOCK()` instead of a transaction is intentional:
1. It works even when the caller is already inside a transaction (nested transactions are not supported in MySQL/InnoDB)
2. Named locks are session-scoped and automatically released if the connection drops
3. The 5-second timeout prevents indefinite blocking

The lock name includes the prefix and year, so different reference number types (LV, LN, ST, EL) don't block each other.

Note: The lock is acquired even if `$got_lock` is false (timeout) — in that case we proceed optimistically since the UNIQUE constraint on the column will catch any collision and the caller can retry. Log a warning if lock acquisition fails:

```php
if ( ! $got_lock ) {
    error_log( sprintf( '[SFS HR] generate_reference_number: failed to acquire lock %s', $lock_name ) );
}
```
