**LV-AUTH-01 — Capability check in handle_approve():**
At the very top of `handle_approve()` (before the existing `check_admin_referer` call at line 1058), add a capability gate:
```php
if ( ! current_user_can( 'sfs_hr.leave.review' ) ) {
    wp_die( __( 'You do not have permission to approve leave requests.', 'sfs-hr' ), 403 );
}
```
This ensures only users with the `sfs_hr.leave.review` capability (granted dynamically to department managers and HR) can reach the handler.

**LV-AUTH-02 — Capability check in handle_cancel():**
At the very top of `handle_cancel()` (before reading `$_POST['id']` at line 1762), add a capability gate. However, handle_cancel() allows the requester to cancel their own request, so the capability check should be more nuanced. Add this early check:
```php
if ( ! current_user_can( 'sfs_hr.leave.review' ) && ! current_user_can( 'sfs_hr.view' ) ) {
    wp_die( __( 'You do not have permission to cancel leave requests.', 'sfs-hr' ), 403 );
}
```
The `sfs_hr.view` capability is granted to all employees (including the requester who wants to cancel their own). The more specific ownership/role check already exists at line 1794. This gate just prevents completely unprivileged users from reaching the handler at all.

**LV-AUTH-03 — Fix is_hr_user() to not grant HR access for bare sfs_hr_manager role:**
In the `is_hr_user()` method (line 2575), remove the check at lines 2580-2582 that returns true for any user with `sfs_hr_manager` role:
```php
// REMOVE these lines:
$user = get_user_by( 'id', $user_id );
if ( $user && in_array( 'sfs_hr_manager', (array) $user->roles, true ) ) {
    return true;
}
```
After removal, `is_hr_user()` should only return true if:
1. The user is in the `sfs_hr_leave_hr_approvers` option array, OR
2. The user has `sfs_hr.leave.manage` capability, OR
3. The user has `administrator` capability

The `sfs_hr_manager` role alone (a department manager with no explicit HR responsibility) should NOT grant HR access.

**LV-AUTH-05 — Prevent HR self-approval in handle_cancellation_approve():**
The existing self-approval check at line 2014 only blocks the leave *requester* (employee) from approving their own cancellation. But an HR user who also filed a leave request can approve their own cancellation at level 2 (HR stage) because the HR check at line 2005-2007 passes before the self-approval check, and the self-approval check compares `empInfo['user_id']` (the leave requester) against `current_uid` — which IS the same user but might pass if a different HR user submitted on their behalf.

The fix is to also check at the HR approval stage (level 2, around line 2071) whether the approving HR user is the same person who requested the leave:
After the `if ( ! $is_hr )` check at line 2071, add:
```php
// Prevent HR self-approval at level 2
if ( (int) ( $empInfo['user_id'] ?? 0 ) === $current_uid ) {
    Helpers::redirect_with_notice( $redirect_base, 'error', __( 'You cannot approve a cancellation for your own leave.', 'sfs-hr' ) );
}
```

Wait — the existing check at line 2014 already covers this before any level check. Let me re-read: the check at line 2014 IS before the level-1/level-2 branching, so it should catch both levels. The issue is more subtle: at level 2, the `$is_hr` check includes `current_user_can('administrator')` — an administrator who is also the employee could reach line 2014 but the check would block them. So the existing check at 2014 IS correct for both levels.

However, re-reading the requirement: "An HR user cannot approve their own leave request at either the first or second approval stage." The current check at line 2014 compares `empInfo['user_id']` — the employee who filed the leave — against the current user. This IS correct. But we need to also handle the case where the HR user who filed the cancellation request (not the original leave, but the cancellation itself) is the same HR user approving it. Check if `cancel['requested_by']` exists and compare:

Actually, looking at the cancellation table, the `requested_by` user_id in the cancellation record is the person who initiated the cancellation. The self-approval prevention should check BOTH:
1. The leave requester (employee) cannot approve their own cancellation (already at line 2014)
2. The person who initiated the cancellation request cannot approve it themselves

Add after the existing check at line 2014:
```php
// Also prevent the cancellation requester from approving their own cancellation request
if ( isset( $cancel['requested_by'] ) && (int) $cancel['requested_by'] === $current_uid ) {
    Helpers::redirect_with_notice( $redirect_base, 'error', __( 'You cannot approve a cancellation you requested.', 'sfs-hr' ) );
}
```

If the cancellations table has no `requested_by` column, check the column structure first (look at the cancellations table CREATE in LeaveModule or Migrations.php). If the column is named differently (e.g., `employee_id` or `created_by`), use that column instead.
