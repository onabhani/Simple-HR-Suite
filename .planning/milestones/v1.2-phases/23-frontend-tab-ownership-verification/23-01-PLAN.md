<task type="auto">
  <name>Task 2: Fix TeamTab role-based visibility for HR/GM/Admin</name>
  <files>includes/Frontend/Tabs/TeamTab.php</files>
  <action>
Fix the hardcoded `$is_manager_only = true` on line 45 so that HR, GM, and Admin roles see all employees across all departments, while managers remain scoped to their managed departments.

**Replace lines 36-44** (the current scope block):
```php
// Determine scope -- always filter by managed departments (direct reports only).
$dept_ids = Role_Resolver::get_manager_dept_ids( $user_id );
if ( empty( $dept_ids ) ) {
    echo '<div class="sfs-empty-state">';
    echo '<p class="sfs-empty-state-title">' . esc_html__( 'No departments assigned.', 'sfs-hr' ) . '</p>';
    echo '<p class="sfs-empty-state-text">' . esc_html__( 'You are not currently managing any departments.', 'sfs-hr' ) . '</p>';
    echo '</div>';
    return;
}
$is_manager_only = true; // Always scope to direct reports
```

**With:**
```php
// Determine scope: HR+ roles (level >= 40) see all employees org-wide;
// managers (level 30) see only their managed departments.
$is_manager_only = ( $level < 40 );

$dept_ids = [];
if ( $is_manager_only ) {
    $dept_ids = Role_Resolver::get_manager_dept_ids( $user_id );
    if ( empty( $dept_ids ) ) {
        echo '<div class="sfs-empty-state">';
        echo '<p class="sfs-empty-state-title">' . esc_html__( 'No departments assigned.', 'sfs-hr' ) . '</p>';
        echo '<p class="sfs-empty-state-text">' . esc_html__( 'You are not currently managing any departments.', 'sfs-hr' ) . '</p>';
        echo '</div>';
        return;
    }
}
```

This change makes the existing dead code on lines 49-53 (the `if ( ! $is_manager_only )` branch that loads all departments) reachable for HR/GM/Admin roles. The rest of the query-building logic already correctly handles the `$is_manager_only` flag:
- When `$is_manager_only = false`: no `dept_id IN (...)` WHERE clause is added, all departments are loaded, the query returns org-wide employees.
- When `$is_manager_only = true`: the existing `dept_id IN (...)` filter remains active.

No other changes needed -- the existing query builder, filters, and rendering code already support both paths.
