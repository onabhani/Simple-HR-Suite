<task type="auto">
  <name>Task 1: Read and audit Assets module orchestrator, REST, and admin controller</name>
  <files>
    includes/Modules/Assets/AssetsModule.php (374 lines -- module bootstrap, hook registration, table creation)
    includes/Modules/Assets/Rest/class-assets-rest.php (23 lines -- REST endpoint registration)
    includes/Modules/Assets/Admin/class-admin-pages.php (2101 lines -- admin CRUD, assignment/return handlers, file uploads)
  </files>
  <action>
Read all three PHP files. Audit each file against the 4 metrics:

**1. Security (Critical focus):**
- Every `$wpdb->query()`, `$wpdb->get_results()`, `$wpdb->get_row()`, `$wpdb->get_var()` call: Is `$wpdb->prepare()` used? Flag any raw string interpolation as Critical.
- AssetsModule table installation/migration: Does it use bare `ALTER TABLE` or unprepared `SHOW TABLES` like Loans/Core did? (Prior phases found this as Critical antipattern -- LOAN-SEC-001/002, Phase 04 Critical.)
- REST permission_callback: Does class-assets-rest.php use `__return_true` for any endpoint? This was Critical in Attendance Phase 05 (GET /attendance/status exposed data to unauthenticated callers). Even at 23 lines, verify every registered route has a real permission callback.
- Admin handler capability checks: Does every admin action handler (create/edit/delete asset, assign/return asset) verify `current_user_can()` with appropriate capability before executing? Flag missing guards as Critical.
- Admin handler nonce validation: Are all write operations protected with `wp_verify_nonce()` or `check_admin_referer()`?
- **File upload handling**: Does the admin controller handle asset photo/document uploads? If so, audit for:
  - MIME type validation (is uploaded file type checked against allowed list, or any file accepted?)
  - Path traversal risk (is the filename sanitized, or can `../../` in filename write outside upload directory?)
  - Direct file access (are uploaded files accessible without authentication via direct URL?)
  - File size limits (any max upload size enforcement?)
- Input sanitization: Are asset IDs, employee IDs, amounts, dates sanitized via `intval()`, `sanitize_text_field()`, or `absint()` before use?
- **Ownership verification**: When an employee views or returns an asset, is there a check that the asset is assigned to THEM? Or can any authenticated user return any asset? Flag missing ownership checks as High.

**2. Performance:**
- Are asset lists loaded without pagination (unbounded `get_results()`)? Flag for companies with many assets.
- Does the admin controller make N+1 queries (e.g., fetching employee name per asset row in a loop)?
- Are there repeated database lookups within the same request?
- Does the assignment list join employees efficiently or loop-query?

**3. Duplication:**
- Is asset status tracked in multiple places (e.g., both a `status` column on the asset AND a separate assignment table with status)? Document any dual-tracking.
- Does the module duplicate employee lookup logic that exists in Core helpers?
- Are there repeated HTML rendering patterns across admin pages that could be shared?

**4. Logical Issues:**
- **Assignment lifecycle**: Can an asset be assigned to multiple employees simultaneously? Is there a check preventing double-assignment?
- **Return logic**: When an asset is returned, is the assignment record updated AND the asset status changed? Or is one forgotten?
- **Status machine**: What are the valid asset statuses and transitions? Are invalid transitions possible (e.g., returning an asset that was never assigned)?
- **Delete safety**: Can an assigned asset be deleted? What happens to the assignment record?
- **Dead code**: Any unused methods or unreachable branches?

Rate every finding: Critical / High / Medium / Low. Include file:line references and a concrete fix recommendation for each.
