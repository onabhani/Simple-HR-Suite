<objective>
Audit the Payroll admin pages (2,576 lines) for security, performance, duplication, and logical issues. This is the largest file in the Payroll module and contains admin UI rendering, payroll run management, payslip generation, export functionality, and payroll report views.

Purpose: Admin_Pages.php is where payroll data is displayed, exported, and managed. Export/report generation is a high-risk area for data leakage (an HR user in one department seeing another department's payroll data) and XSS (unescaped employee names or amounts in CSV/HTML output). The payroll calculation display must handle floating-point amounts correctly. This file likely contains the bulk of the $wpdb queries in the Payroll module.

  <action>
Read Admin_Pages.php (2,576 lines). Audit against the 4 metrics:

**1. Security (Critical focus):**
- Every `$wpdb->query()`, `$wpdb->get_results()`, `$wpdb->get_row()`, `$wpdb->get_var()` call: Is `$wpdb->prepare()` used? Flag raw interpolation as Critical.
- Admin page access: Does every admin page callback check `current_user_can('sfs_hr.manage')` or equivalent before rendering payroll data?
- Write operations (save payroll run, update payslip, mark as paid): Do they verify nonce + capability?
- Export handlers (CSV, Excel, PDF export): Are they protected with capability checks? Can a URL be shared/bookmarked to bypass auth?
- Output escaping: Is employee data (names, department, amounts) escaped with `esc_html()` / `esc_attr()` when rendered in admin HTML? Are amounts sanitized in export output?
- Nonces in HTML attributes: Check for the same antipattern found in Loans (LADM-SEC-002: nonce in data-nonce HTML attribute readable by scripts).

**2. Performance:**
- Payroll list/summary views: Is there pagination? Or does it load all payroll records for all periods? (Same antipattern as LADM-PERF-001 in Loans.)
- Per-employee payslip generation: Does it trigger N+1 queries (one query per employee for deductions, allowances, loan amounts)?
- Report generation: Are aggregate queries efficient (single GROUP BY) or do they loop over employees?
- Are there unbounded JOINs on large tables (employees x payroll_runs x deductions)?

**3. Duplication:**
- Salary component calculation: Does the admin page recalculate salary breakdown (basic + allowances - deductions) independently from PayrollModule.php? Compare formulas.
- Export formatting: Is there repeated column/header formatting across CSV, Excel, and print views?
- Employee data fetching: Are there duplicate queries for employee details that could share a single fetch?
- Date/period formatting: Are payroll period labels generated in multiple places?

**4. Logical Issues:**
- Floating-point display: Are payroll amounts displayed with consistent decimal precision? Check for `number_format()` or `round()` usage.
- Payroll run status display: Does the UI correctly reflect all possible states (draft, processing, completed, cancelled)?
- Department filtering: Can a department manager see payroll data for other departments? Is there row-level filtering by department?
- Payslip completeness: Does the payslip view include all components (gross, deductions, net) or could components be silently missing?
- Export data integrity: Does the exported data match what is shown on screen? Are there off-by-one issues with date ranges in filtered exports?
- Currency handling: Are amounts formatted assuming SAR? Is there any currency mismatch risk?

Rate every finding: Critical / High / Medium / Low. Include file:line references and a concrete fix recommendation for each.
