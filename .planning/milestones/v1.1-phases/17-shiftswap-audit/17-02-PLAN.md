  <action>
Read both files completely. Audit against the 4 metrics:

**1. Security (Critical focus -- admin auth and REST permissions):**
- class-shiftswap-tab.php -- Capability gates:
  - Is there an outer current_user_can() check at the top of the render method? Or can any authenticated user with admin page access view all swap requests? (Same pattern as Documents Phase 16 DADM-SEC-001, Resignation Phase 14 RES-SEC-001)
  - Which capability is used? Is it the correct capability (sfs_hr.manage or equivalent) or a wrong/weak one (sfs_hr_attendance_view_team pattern from Phase 15/16)?
  - Are action buttons (approve/reject) shown without capability check in the view? (Same pattern as Settlement Phase 10 SADM-SEC-001)
- class-shiftswap-tab.php -- Output escaping:
  - Are employee names, shift details, and other dynamic data escaped with esc_html(), esc_attr() before rendering?
  - Are URLs escaped with esc_url()?
  - Any raw echo of database values without escaping = High XSS finding
- class-shiftswap-tab.php -- $wpdb queries:
  - Catalogue every $wpdb call with file:line
  - Check each for $wpdb->prepare() usage
  - Check for inline queries in the view (architectural violation, same as Assets Phase 11 AVIEW-SEC-001/002)
- class-shiftswap-rest.php -- Permission callbacks:
  - Does each registered route have a permission_callback?
  - Is any route using __return_true? (Critical if found -- same as Attendance Phase 05)
  - What capabilities are checked in permission callbacks?
  - If the file is a stub with no routes registered, note that (acceptable pattern, same as Assets Phase 11)

**2. Performance:**
- Check for N+1 patterns in admin list rendering (e.g., fetching employee name per row in a loop)
- Check for unbounded queries (no LIMIT/pagination on swap request listing)
- Check for information_schema or SHOW TABLES on page load

**3. Duplication:**
- Check if admin tab list rendering duplicates patterns that could use a shared admin table helper
- Check if approval/rejection action handling in the view duplicates handler logic

**4. Logical Issues:**
- Check if admin tab correctly filters by department for department managers (or shows all-org data to non-admin users -- same as Assets Phase 11 AVIEW-LOGIC-001)
- Check if swap request status display reflects actual state machine
- Check for dead/unreachable code paths

**For every $wpdb call found:**
- Catalogue with file:line reference
- Mark as "prepared" or "raw"
- If raw: static (safe but antipattern) or dynamic (injection risk)
- Rate severity

**Output format:** Create findings report following established pattern with sections: Security Findings, Performance Findings, Duplication Findings, Logical Issues, $wpdb Call Accounting table, Summary table with severity counts.
