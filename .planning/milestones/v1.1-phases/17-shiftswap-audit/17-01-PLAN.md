  <action>
Read all 4 files completely. Audit against the 4 metrics:

**1. Security (Critical focus -- swap ownership and approval auth):**
- ShiftSwapModule.php: Check for bare ALTER TABLE, unprepared SHOW TABLES, information_schema queries (antipattern from Phase 04/08/11/16). Check install()/activation hooks for DDL safety.
- ShiftSwap_Service.php -- Swap request creation:
  - When creating a swap request, does the code verify the requesting employee OWNS the source shift? Or can any employee create a swap request for another employee's shift?
  - Is the target employee validated as a real employee with a compatible shift?
  - Is there a check preventing duplicate/overlapping swap requests for the same shift?
- ShiftSwap_Service.php -- Approval logic:
  - Does the approval method check that the approver has manager capability (sfs_hr.manage or department manager dynamic cap)?
  - Can the requesting employee approve their own swap request?
  - Is there a state machine guard preventing approved->pending backwards transitions?
- ShiftSwap_Handlers.php:
  - Does each handler have nonce verification (check_admin_referer or wp_verify_nonce)?
  - Does each handler have capability checks (current_user_can)?
  - Are POST parameters sanitized before use?
- ShiftSwap_Notifications.php:
  - Are notification recipients correctly scoped (only involved employees + their manager)?
  - Is any sensitive shift/employee data exposed in notification content that shouldn't be?

**2. Performance:**
- Look for N+1 query patterns (e.g., fetching employee/shift details inside loops)
- Check for unbounded queries (SELECT without LIMIT on swap request listings)
- Check for information_schema or SHOW TABLES queries on every page load

**3. Duplication:**
- Check if shift ownership validation logic duplicates Attendance module shift resolution
- Check if notification sending logic duplicates patterns from other modules (e.g., uses dofs_ prefixed filters like Resignation Phase 14)
- Check if approval workflow duplicates Leave or Resignation approval patterns without shared helper

**4. Logical Issues:**
- Check for TOCTOU race conditions on approval (unconditional UPDATE without WHERE status=expected -- same pattern as Resignation Phase 14, Hiring Phase 13)
- Check for dead code or unreachable branches
- Verify swap execution actually updates both employees' shift assignments (not just marking request as approved)

**For every $wpdb call found:**
- Catalogue the call with file:line reference
- Mark as "prepared" (uses $wpdb->prepare()) or "raw" (string interpolation/concatenation)
- If raw: determine if static (safe but antipattern) or dynamic (injection risk)
- Rate severity: Critical for dynamic raw, Medium for static raw

**Output format:** Create findings report following the established pattern from prior phases (Phase 16, 15, etc.) with sections: Security Findings, Performance Findings, Duplication Findings, Logical Issues, $wpdb Call Accounting table, Summary table with severity counts.
