  <files>
    includes/Modules/Workforce_Status/Cron/Absent_Cron.php (161 lines -- cron scheduling, manual trigger, status reporting)
    includes/Modules/Workforce_Status/Notifications/Absent_Notifications.php (503 lines -- absent employee detection, manager/employee email notifications)
  </files>
  <action>
Read both files completely. Audit against the 4 metrics:

**1. Security:**
- Catalogue every `$wpdb` call in Absent_Notifications.php: Lines 32-45 (employees query), 67-75 (clocked-in query), 79-88 (leave query). Verify all use `$wpdb->prepare()`.
- `trigger_manual()` (Absent_Cron line 133): This is a public static method that sends notifications for any date. Is it exposed via any admin-post handler or REST endpoint? If so, does it have a capability check? If not directly exposed, flag as informational.
- `send_absent_notifications()` and `send_employee_absent_notifications()`: Do they validate the `$date` parameter format? Could a malformed date cause SQL injection or unexpected behavior?
- `update_settings()` (line 500-502): Accepts any array and saves directly to wp_options. Is this called from an admin handler with sanitization? Check if the handler validates/sanitizes before calling this.
- Email content: `build_notification_message()` and `build_employee_absent_message()` -- are employee names and department names escaped in HTML output? (Yes -- check esc_html usage throughout the templates.)
- `$wpdb->prepare()` with empty params array on line 43: `$wpdb->prepare( "...", [] )` -- verify WordPress handles this correctly (it should since there are no placeholders in the base query, but the prepare call is unnecessary).

**2. Performance:**
- `get_absent_employees_by_department()` loads ALL active employees (no LIMIT) on every cron run. On a 500+ employee org, this is 3 queries touching all employees. Flag severity.
- `is_employee_day_off()` calls `AttendanceModule::resolve_shift_for_date()` once per employee (line 203) -- same N+1 pattern as Admin_Pages. Flag as High.
- `send_employee_absent_notifications()` calls `get_absent_employees_by_department()` again (line 360), duplicating the exact same 3 queries that `send_absent_notifications()` already ran on line 217. The cron `run()` method calls both sequentially (lines 105-106), meaning the absent employee detection runs TWICE per cron execution. Flag as High.
- No batch email sending -- each absent employee gets an individual `Helpers::send_mail()` call. On 50 absent employees, that is 50 sequential SMTP connections. Flag as Medium.

**3. Duplication:**
- `is_holiday()` (lines 142-175) is nearly identical to `Admin_Pages::is_holiday()`. Extract to shared helper.
- `is_employee_day_off()` (lines 184-208) overlaps with `Admin_Pages::is_working_day_for_employee()` and `get_employee_shifts_map()` but with different implementation (Notifications checks `weekly_off_days` setting; Admin_Pages fallback only checks Friday).
- `get_absent_employees_by_department()` duplicates query logic that could be shared with Admin_Pages `get_all_rows()` -- both query employees, punches, and leave for the same date.

**4. Logical issues:**
- `is_employee_day_off()` checks `weekly_off_days` setting and defaults to `[5]` (Friday only). Saudi weekend is Friday + Saturday. This should default to `[5, 6]` or read from a setting that includes Saturday. Flag as High (same issue as Phase 09 PAY-LOGIC-001).
- `$wpdb->prepare()` with empty array on line 43-44: The base employees query has no `%s`/`%d` placeholders but is wrapped in prepare(). This triggers a WordPress deprecation notice in WP 6.x+ ("wpdb::prepare was called incorrectly. The query argument does not contain a placeholder."). Flag as Medium.
- Cron `run()` at line 96-106: If `send_absent_notifications()` fails (throws exception), `send_employee_absent_notifications()` never runs. No try/catch isolation.
- `get_next_run_timestamp()`: If `DateTimeImmutable::createFromFormat()` returns false AND `$target <= $now` is never evaluated because of short-circuit, the fallback `$now->modify('+1 day')->setTime(20,0)` is used. This is correct but the null-coalescing logic is subtle.
- `send_employee_absent_notifications()` hardcodes `home_url('/my-profile/?sfs_hr_tab=leave')` as the leave URL. If the site uses a different page for the profile shortcode, this link is broken. Flag as Low.

For each finding, assign severity (Critical/High/Medium/Low) and provide a concrete fix recommendation.
