  <files>
    includes/Modules/Workforce_Status/WorkforceStatusModule.php (33 lines -- module bootstrap, hook registration, no DDL)
    includes/Modules/Workforce_Status/Admin/Admin_Pages.php (1335 lines -- workforce status dashboard with tabs, filters, queries, risk flags)

  <action>
Read all 3 files completely. Audit against the 4 metrics:

**1. Security (Critical focus):**
- Catalogue every `$wpdb` call in Admin_Pages.php: Is `$wpdb->prepare()` used for all? Note lines 813-816 where `get_department_map()` runs an unprepared query when `$allowed_depts` is null (all-departments case for sfs_hr.manage users). Flag severity.
- Does `render_page()` have a capability gate at entry? (Yes -- line 38 `Helpers::require_cap('sfs_hr_attendance_view_team')`). Confirm this is the right capability.
- Department scoping: When `current_user_can('sfs_hr.manage')` is false, does `manager_dept_ids_for_current_user()` correctly restrict data? What if the returned array is empty -- is the user blocked?
- Output escaping: Are all echoed values properly escaped? Check `$page_links` on line 647 (paginate_links output), `render_status_badge()`, `render_leave_badge()`, and the JSON data on line 698.
- `$_GET` params: Are `tab`, `paged`, `per_page`, `s`, `dept` all sanitized? Check each.
- WorkforceStatusModule.php: No DDL/install() method -- confirm no bare ALTER TABLE or SHOW TABLES.
- Status_Analytics.php: Empty file -- flag as dead code.

**2. Performance:**
- `get_all_rows()` loads ALL active employees from DB (no LIMIT), then filters and paginates in PHP. On a 500+ employee org, this means loading all employees, all punches, all leave, all risk sessions on every page view. Flag severity.
- `get_employee_shifts_map()` calls `AttendanceModule::resolve_shift_for_date()` once per employee in a loop (line 1183). This is N+1 if the underlying method does a DB query. Flag severity.
- `is_holiday()` is called twice per page view: once at line 880 for global check, and again inside `is_working_day_for_employee()` at line 1257 for EVERY employee. Flag the redundant calls.
- `get_risk_flags_map()` loads all sessions for all employees for 30 days. On 500 employees with 30 days = potentially 15K rows. Check if this is bounded.
- `get_today_punch_map()` uses ORDER BY punch_time ASC and overwrites map entries (last row wins) -- effectively doing MAX(punch_time) in PHP instead of SQL. Flag as medium performance issue.

**3. Duplication:**
- `is_holiday()` in Admin_Pages.php (lines 1126-1163) is nearly identical to `is_holiday()` in Absent_Notifications.php (lines 142-175). Flag as duplication.
- `manager_dept_ids_for_current_user()` (lines 1317-1334) is noted as "copy of Core\Admin::manager_dept_ids()". Flag this acknowledged duplication.
- Department scoping pattern (allowed_depts + placeholders) is repeated across `get_department_map()`, `get_all_rows()`, and notification queries.

**4. Logical issues:**
- `is_friday()` fallback (line 1221-1225) only checks Friday but Saudi weekend is Friday + Saturday. The `is_employee_day_off()` in Notifications checks `weekly_off_days` setting but Admin_Pages fallback does not. Flag inconsistency.
- `compute_status_from_punch()`: break_end maps to clocked_in which is correct, but if an employee punches break_start then never punches break_end, they show as "on_break" indefinitely. Document as known limitation.
- Risk flag thresholds are hardcoded constants (5 days each) with no admin UI to configure. Flag as Low.
- `get_today_punch_map()` keeps last punch per employee. If last punch is "out" at 5pm, employee correctly shows as clocked_out. But if employee punches "in" again after hours (second shift), the earlier "out" is lost. Flag as edge case.

For each finding, assign severity (Critical/High/Medium/Low) and provide a concrete fix recommendation.
