  <action>
Read the entire class-employee-profile-page.php file. Audit against the 4 metrics:

**1. Security (Critical focus):**
- Catalogue every `$wpdb` call (all 80): Is `$wpdb->prepare()` used? Flag any raw string interpolation as Critical.
- Capability checks: Does the menu registration use an appropriate capability? Does every write handler (save, delete, create WP user) verify `current_user_can()` with a specific capability before executing?
- Nonce validation: Are all POST/action handlers protected with `check_admin_referer()` or `wp_verify_nonce()`?
- WP user creation handler (line ~39-80+): Audit for privilege escalation -- can a low-privilege user trigger user creation? Is the role assigned to the new user appropriate? Is the password handling secure?
- Output escaping: Are employee fields (name, email, phone, address, notes) properly escaped with `esc_html()`, `esc_attr()` when rendered in HTML? Flag unescaped output as High.
- Input sanitization: Are employee IDs, form fields sanitized before use in queries or output?

**2. Performance:**
- N+1 query patterns: Does profile rendering execute queries inside loops (e.g., fetching department name per employee, attendance per day)?
- Unbounded queries: Any `SELECT *` without LIMIT on potentially large tables?
- Heavy operations on page load: Are KPI risk indicator calculations (RISK_LOOKBACK_DAYS constants at lines 16-19) doing expensive queries on every profile view?

**3. Duplication:**
- Repeated query patterns: Are similar SELECT queries duplicated across methods?
- Date formatting logic: Is hire_date/hired_at formatting duplicated?
- HTML rendering patterns: Are card/badge rendering patterns repeated without helpers?

**4. Logical issues:**
- Dual hire_date/hired_at: Line 175 uses `$emp['hired_at'] ?? ( $emp['hire_date'] ?? '' )` -- is this fallback used consistently across all methods, or do some methods use only one column?
- KPI risk calculation: Are the thresholds (RISK_LATE_MIN_DAYS=3, etc.) correctly applied? Any off-by-one or timezone issues?
- Employee status transitions: Are there any dead code paths or unreachable conditions?
- Race conditions: Can concurrent profile edits corrupt data?

For each finding, assign severity (Critical/High/Medium/Low) and provide a concrete fix recommendation.
