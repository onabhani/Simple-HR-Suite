  <action>
Read every PHP file listed above. For each file, check against the following 4 audit metrics and record every finding:

**Security (Critical/High):**
- SQL injection: Every `$wpdb->query()`, `$wpdb->get_results()`, `$wpdb->get_var()`, `$wpdb->get_row()` call -- is the query built with `$wpdb->prepare()` or is user input concatenated/interpolated directly? Flag raw interpolation as Critical.
- Auth checks in cron: Cron callbacks should not process external input, but verify they do not accidentally expose endpoints or skip validation. Flag as High if found.
- Data sanitization: Any user input from `$_POST`, `$_GET`, `$_REQUEST` used without sanitization. Flag as High.

**Performance (Medium/High):**
- N+1 queries: Loops that execute a DB query per iteration (e.g., looping employees to build sessions individually). Flag as High.
- Unbounded queries: `SELECT` without `LIMIT` on tables that grow (sessions, attendance logs). Flag as Medium.
- Redundant DB calls: Same query executed multiple times without caching result. Flag as Medium.
- Cron efficiency: Cron jobs that process all employees without batching or that run expensive queries. Flag as Medium.

**Duplication (Medium/Low):**
- Repeated shift/period calculation logic across services. Flag as Medium and note which services share the pattern.
- Date/time helper logic duplicated instead of using a shared utility. Flag as Low.
- Employee lookup patterns repeated across services. Flag as Low.

**Logical Issues (varies):**
- Race conditions in Session_Service: Punch-in/out logic that reads current session then writes without locking -- concurrent requests could create overlapping sessions. Flag as Critical if data corruption possible, High otherwise.
- Timezone handling: Are shift times compared against punch times in consistent timezones? Off-by-hour bugs possible. Flag as High if inconsistent.
- Off-day/overtime edge cases: Does session builder handle employees working on off-days or overtime correctly? Flag as Medium.
- Migration idempotency: Are all CREATE TABLE and ALTER statements safe to re-run? Flag as High if not idempotent.
- Dead code: Methods never called anywhere. Flag as Low.

For each finding, record:
1. **Severity**: Critical / High / Medium / Low
2. **Category**: Security, Performance, Duplication, or Logic
3. **File and line number(s)**: e.g., `includes/Modules/Attendance/Services/Session_Service.php:L245-L260`
4. **Description**: What the issue is, in one sentence
5. **Evidence**: The problematic code snippet (1-5 lines)
6. **Fix recommendation**: Concrete fix in one sentence

Start with Session_Service.php (highest risk -- punch logic and race conditions), then Shift_Service.php, Policy_Service.php, then remaining files.
