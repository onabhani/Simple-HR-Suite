  <action>
Read the entire class-admin-pages.php file. Audit against the 4 metrics:

**1. Security (Critical focus):**
- Catalogue every `$wpdb` call (all 54): Is `$wpdb->prepare()` used? Flag any raw string interpolation as Critical.
- Menu capability: What capability does `add_submenu_page()` use? Is it appropriate for hiring management or too broad/narrow?
- Candidate list search: Line ~137 uses `$wpdb->esc_like($search)` -- is the search parameter also passed through `$wpdb->prepare()` with `%s` placeholder, or is it concatenated after esc_like? (esc_like alone is insufficient)
- Conditional prepare pattern: Lines ~149 and ~779 use `$params ? $wpdb->get_results($wpdb->prepare(...)) : $wpdb->get_results($query)` -- when $params is empty, the raw $query runs without prepare. Is this safe (static query) or dangerous (dynamic parts)?
- Pipeline approval capability: Lines ~388-389 use `current_user_can('manage_options')` for dept/GM approval -- should use HR-specific capabilities like `sfs_hr.manage` or `sfs_hr.hiring.approve`
- POST handler nonce checks: Do `handle_add_candidate()`, `handle_update_candidate()`, `handle_candidate_action()` and trainee equivalents all verify nonces? Are the nonce action names unique per operation?
- POST handler capability checks: After nonce verification, do handlers also check `current_user_can()` before executing writes? (nonce alone is insufficient)
- Output escaping: Are candidate/trainee fields (name, email, phone, notes, reference codes) escaped with `esc_html()` / `esc_attr()` when rendered in HTML? Check both list views and detail views.
- Trainee evaluation form: Does it accept and store free-text evaluation data? Is it sanitized?

**2. Performance:**
- Candidate list: Is there pagination, or are all candidates loaded at once? (Line ~149 query has no LIMIT)
- Trainee list: Same pagination check (line ~779)
- Department/employee dropdowns: Lines ~248, 875-876 load all departments and all active employees for dropdown selects -- on large orgs this could be expensive
- N+1 patterns: Do list renderings execute per-row queries (e.g., fetching department name per candidate when JOIN could be used)?

**3. Duplication:**
- Candidate vs Trainee patterns: render_candidates_list() vs render_trainees_list(), render_candidate_form() vs render_trainee_form(), render_candidate_view() vs render_trainee_view() -- document overlap percentage and recommend shared abstractions
- Nonce field repetition: Multiple `wp_nonce_field('sfs_hr_candidate_action')` calls in the candidate view -- is the same nonce reused for different actions (approve, reject, offer, hire)?
- Status badge rendering: Are candidate/trainee status badges rendered with duplicate HTML/CSS patterns?
- CRUD handler patterns: Are add/update handlers for candidates structurally identical to trainee handlers?

**4. Logical issues:**
- Approval chain state machine: Does handle_candidate_action() enforce a valid state transition order (new -> dept_approved -> gm_approved -> offer_sent -> hired)? Can stages be skipped or reversed?
- Double-action prevention: Can the same approval action be submitted twice (e.g., two GMs clicking approve simultaneously)?
- Rejected candidate re-activation: Can a rejected candidate be moved back to an active pipeline stage?
- Trainee status transitions: Are trainee statuses (active, completed, terminated, converted) enforced with valid transition rules?
- Tab routing: Does render_page() safely handle unknown tab values? What about unknown action values within a tab?

For each finding, assign severity (Critical/High/Medium/Low) and provide a concrete fix recommendation.
