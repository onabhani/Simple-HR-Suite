  <action>
Read the entire HiringModule.php file. Audit against the 4 metrics:

**1. Security (Critical focus):**
- Catalogue every `$wpdb` call (all 44): Is `$wpdb->prepare()` used? Flag any raw string interpolation as Critical.
- install() DDL: Does it use bare ALTER TABLE or unprepared SHOW TABLES / information_schema? (Critical antipattern from Phase 04, 08, 11)
- Conversion methods capability checks: Do `convert_trainee_to_candidate()`, `convert_trainee_to_employee()`, and `convert_candidate_to_employee()` verify `current_user_can()` before creating employee records, WP users, and assigning roles? These are public static methods -- can any caller invoke them?
- WP user creation in conversion: When a trainee/candidate is converted to employee, a WP user is created with a role. Is the role safe (subscriber/sfs_hr_employee)? Is the password generation secure? Is `send_welcome_email()` leaking credentials in a recoverable way?
- Employee code generation race condition: `generate_candidate_reference()` and `generate_trainee_code()` read the last code then increment -- is there a TOCTOU race where concurrent calls get the same code? No UNIQUE constraint visible in CREATE TABLE?

**2. Performance:**
- install() on activation: Are the CREATE TABLE IF NOT EXISTS statements efficient? Any unnecessary column types?
- Code generation: Does `generate_trainee_code()` / `generate_candidate_reference()` do an ORDER BY + LIMIT 1 query -- efficient enough, or is there a sequence table pattern that would be better?
- Conversion methods: Do they do multiple sequential queries that could be batched? Are there N+1 patterns?

**3. Duplication:**
- `convert_trainee_to_employee()` vs `convert_candidate_to_employee()`: These two methods likely share significant logic (create employee record, create WP user, assign role, send email). Document the overlap percentage and recommend extraction.
- Code generation: Are `generate_candidate_reference()` and `generate_trainee_code()` essentially the same pattern with different table/column names?
- Notification methods: Do `notify_hr_trainee_hired()` and `notify_hr_new_hire()` duplicate email composition logic?

**4. Logical issues:**
- Conversion atomicity: Are the multi-step conversions (read source record, create employee, create WP user, update source status) wrapped in a transaction? What happens if the WP user creation fails after the employee record is inserted?
- Status transitions: When converting, is the source record's status updated atomically with the new record creation? Can a record be converted twice?
- Employee code format: Is the code format consistent between trainee-originated and candidate-originated employees?
- Trainee supervisor_id: Does it reference sfs_hr_employees.id or WP user_id? Is the FK relationship correct?

For each finding, assign severity (Critical/High/Medium/Low) and provide a concrete fix recommendation.
