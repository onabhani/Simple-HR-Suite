<objective>
Audit the Loans module orchestrator, frontend loan request handler, and notifications for security, performance, duplication, and logical issues. This plan covers the employee-facing layer and module bootstrap: loan request submission (admin_post + wp_ajax), installment schedule rendering, notification triggers, and table installation.

Purpose: The Loans module handles financial operations where calculation errors directly affect employee pay. The frontend handler (810 lines) processes loan requests and renders installment schedules -- missing auth or nonce checks here would allow unauthorized loan creation. The notifications layer (720 lines) sends loan-related alerts that may leak financial data. The module orchestrator (537 lines) bootstraps tables and hooks.

<task type="auto">
  <name>Task 1: Read and audit Loans module orchestrator, frontend, and notifications</name>
  <files>
    includes/Modules/Loans/LoansModule.php (537 lines -- module bootstrap, table creation, hook registration)
    includes/Modules/Loans/Frontend/class-my-profile-loans.php (810 lines -- employee loan request form, installment schedule display, admin_post + wp_ajax handlers)
    includes/Modules/Loans/class-notifications.php (720 lines -- loan approval/rejection/disbursement notifications)
  </files>
  <action>
Read all 3 PHP files. Audit each file against the 4 metrics:

**1. Security (Critical focus):**
- Every `$wpdb->query()`, `$wpdb->get_results()`, `$wpdb->get_row()`, `$wpdb->get_var()` call: Is `$wpdb->prepare()` used? Flag any raw string interpolation as Critical.
- Frontend handlers (`handle_loan_request`, `handle_loan_request_ajax`): Is `wp_verify_nonce()` checked before processing? Is the employee ID derived from the current user (not from POST data an attacker could manipulate)?
- Can an employee submit a loan request on behalf of another employee (IDOR)? Check if employee_id comes from `$_POST` vs `get_current_user_id()`.
- Notifications: Are loan amounts, approval details, or other sensitive financial data included in notification content that could be intercepted (email plaintext)?
- LoansModule table installation action (`admin_post_sfs_hr_install_loans_tables`): Is it protected with capability check and nonce?

**2. Performance:**
- Frontend installment schedule rendering: Does it query all installments without pagination for employees with many loans?
- Are there N+1 patterns where loan list queries trigger per-loan sub-queries for installment counts or payment status?
- Notification queries: Are they bounded? Could a bulk loan operation trigger unbounded notification queries?

**3. Duplication:**
- Installment calculation: Is there schedule-generation logic in the frontend that duplicates logic in Admin pages? Compare installment amount formulas.
- Does notification content formatting duplicate display logic from the frontend tab?
- Are there repeated loan status check patterns that could be a shared helper?
- Cross-module: Does installment deduction logic duplicate what Payroll uses when calculating loan_deduction? (Payroll references `loan_deduction` column -- check if Loans recalculates this independently.)

**4. Logical Issues:**
- Installment arithmetic: What happens with rounding on installment amounts? If loan_amount / num_installments has a remainder, does the last installment absorb it or is money lost/created?
- Early payoff: Is there logic for paying off a loan early? If so, does it correctly recalculate remaining balance?
- Partial months: If a loan starts mid-month, is the first installment prorated or full?
- Zero/negative amounts: Can a loan be created with zero or negative amount? Zero installments?
- Concurrent requests: Can an employee submit multiple loan requests simultaneously (race condition on balance/eligibility check)?
- Loan status transitions: Are there valid state transitions (pending -> approved -> disbursed -> repaying -> completed)? Can states be skipped or reversed?

Rate every finding: Critical / High / Medium / Low. Include file:line references and a concrete fix recommendation for each.
