<objective>
Audit the Settlement module's admin layer -- the admin controller and three view classes -- for security, performance, duplication, and logical issues. This plan covers Settlement_Admin.php (24 lines -- admin page registration/routing), Settlement_Form.php (230 lines -- create/edit settlement form), Settlement_List.php (376 lines -- settlement list table), and Settlement_View.php (425 lines -- single settlement detail view with approval actions).

Purpose: The admin views render financial settlement data and accept user input for creating/editing settlements. Unescaped output in financial views risks XSS. Missing input validation on settlement amounts could allow data corruption. The list view loading all settlements without pagination is a performance risk for organizations with many terminated employees. Prior phases found Critical unprepared SQL in admin views (Phase 04 Frontend, Phase 09 Admin_Pages) -- this audit checks whether Settlement admin has the same issues.
Output: `.planning/phases/10-settlement-audit/10-02-settlement-admin-findings.md`
</objective>

<task type="auto">
  <name>Task 1: Read and audit Settlement admin controller and view classes</name>
  <files>
    includes/Modules/Settlement/Admin/class-settlement-admin.php (24 lines -- admin page registration and routing)
    includes/Modules/Settlement/Admin/Views/class-settlement-form.php (230 lines -- create/edit settlement form rendering)
    includes/Modules/Settlement/Admin/Views/class-settlement-list.php (376 lines -- settlement list table rendering)
    includes/Modules/Settlement/Admin/Views/class-settlement-view.php (425 lines -- single settlement detail view, approval actions)
  </files>
  <action>
Read all four PHP files. Audit each file against the 4 metrics:

**1. Security (Critical focus):**
- Every `$wpdb->query()`, `$wpdb->get_results()`, `$wpdb->get_row()`, `$wpdb->get_var()` call: Is `$wpdb->prepare()` used? Flag any raw string interpolation as Critical.
- Output escaping: Is every dynamic value rendered in HTML escaped with `esc_html()`, `esc_attr()`, `esc_url()`, or `wp_kses()`? Flag unescaped output as High (XSS risk in financial data views).
- Form nonce fields: Do create/edit forms include `wp_nonce_field()`? Flag missing nonces as Critical.
- Capability checks: Do admin page callbacks verify `current_user_can()` before rendering? Flag missing checks as High.
- Input handling: Are `$_GET`/`$_POST`/`$_REQUEST` values sanitized before use in queries or output?

**2. Performance:**
- List view: Does Settlement_List load all settlements or use pagination (LIMIT/OFFSET)? Flag unbounded queries as Medium for organizations with 100+ historical settlements.
- Does the list view make per-row sub-queries (e.g., fetching employee name for each settlement row)? Count N+1 patterns.
- Does the detail view load related data (employee info, salary components, approval history) in separate queries that could be JOINed?

**3. Duplication:**
- Do the view classes duplicate settlement status logic (mapping status codes to labels/colors) that could be a shared helper?
- Is the employee lookup (converting employee_code to name/details) duplicated across form, list, and view classes?
- Are there repeated HTML patterns (status badges, salary component tables) that are copy-pasted between views?
- Compare with Payroll admin views (Phase 09 found components_json parsing duplicated 6 times -- PADM-DUP-001). Does Settlement have similar duplication?

**4. Logical Issues:**
- Form validation: Can a settlement be created with zero or negative amounts? Is there client-side or server-side validation?
- Status display: Does the view accurately reflect all possible settlement states (draft, pending_approval, approved, paid, cancelled)?
- Approval actions: Does the view show approve/reject buttons only to users with appropriate capability? Or are buttons shown to everyone with server-side rejection?
- Currency formatting: Are financial amounts displayed consistently (decimal places, SAR symbol)?
- Date display: Are dates formatted consistently and timezone-aware?

Rate every finding: Critical / High / Medium / Low. Include file:line references and a concrete fix recommendation for each.
  </action>
