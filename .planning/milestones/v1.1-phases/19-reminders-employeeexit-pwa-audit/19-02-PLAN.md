Read PWAModule.php completely. Audit against the 4 metrics:

**1. Security (Critical focus -- service worker data leakage and push subscription auth):**

- Check for bare ALTER TABLE, unprepared SHOW TABLES, information_schema queries (recurring antipattern from Phase 04/08/11/12/16/18).
- Service worker endpoint: Does the generated service worker JS cache API responses that contain employee data (names, codes, attendance, leave balances)? Cached responses in service workers are accessible to any script on the same origin.
- Service worker scope: Is the service worker scoped narrowly to the HR portal or does it intercept all site requests?
- Manifest endpoint: Does the manifest JSON include internal URLs, API endpoints, or admin paths that could aid reconnaissance?
- Push notification subscription: Is the subscription registration endpoint authenticated? Can unauthenticated users register push endpoints and receive employee notifications?
- Does the module serve the service worker JS inline or as a file? If inline via wp_ajax/REST, is the Content-Type set correctly (application/javascript)?
- Are there any REST endpoints with __return_true permission callbacks?
- Any VAPID keys or push credentials hardcoded or stored insecurely?
- Are all $wpdb queries using $wpdb->prepare()? Catalogue every raw interpolation.

**2. Performance:**
- Does the service worker cache strategy cause stale data issues (cache-first vs network-first for dynamic HR data)?
- Are manifest/service worker endpoints generating content on every request or properly cached?
- Any heavy queries triggered by service worker registration or push subscription?

**3. Duplication:**
- Does PWA duplicate notification dispatch logic from Core/Notifications or Reminders module?
- Does PWA duplicate auth/session checking logic that exists elsewhere?
- Any repeated patterns that could use shared helpers?

**4. Logical Issues:**
- Module status: CLAUDE.md marks PWA as "Stub / incomplete" -- identify dead code, unfinished features, TODO comments, methods that exist but do nothing.
- Offline support: If offline pages are served, do they correctly handle stale auth state (user logged out but cached page still shows HR data)?
- Push subscription lifecycle: Are stale subscriptions cleaned up? If an employee is terminated, are their push subscriptions removed?
- Service worker update: Is there a versioning strategy for the service worker, or will old cached versions persist indefinitely?
- Race conditions: Any check-then-act patterns without DB-level guards?
