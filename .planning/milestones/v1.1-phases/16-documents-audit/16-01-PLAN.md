<task type="auto">
  <name>Task 1: Read and audit DocumentsModule orchestrator, service, and handlers</name>
  <files>
    includes/Modules/Documents/DocumentsModule.php (146 lines -- module bootstrap, hook registration, possible DDL)
    includes/Modules/Documents/Services/class-documents-service.php (647 lines -- document CRUD, file upload, storage, retrieval)
    includes/Modules/Documents/Handlers/class-documents-handlers.php (367 lines -- form handlers for document operations)
  </files>
  <action>
Read all 3 files completely. Audit against the 4 metrics:

**1. Security (Critical focus -- document upload is high-risk):**
- DocumentsModule.php: Check for bare ALTER TABLE, unprepared SHOW TABLES, information_schema queries (antipattern from Phase 04/08/11). Check install()/activation hooks for DDL safety.
- Documents_Service.php -- File upload security:
  - Is MIME type validated against an allowlist before saving? Or does it accept any uploaded file type (SVG/HTML injection risk, same as Assets Phase 11 ASSET-SEC-002)?
  - Is file size validated server-side before processing? Or does it rely solely on client-side limits?
  - Are uploaded files stored inside `wp-content/uploads/` with `.htaccess` protection, or in a publicly accessible directory?
  - Is `wp_handle_upload()` or similar WP function used, or does the code use raw `move_uploaded_file()`?
  - Path traversal: Can a crafted filename escape the upload directory?
  - Direct file access: Can an unauthenticated user access uploaded documents by guessing/enumerating URLs?
- Documents_Service.php -- Access control:
  - When retrieving a document, is the requesting user's employee_id checked against the document's owner?
  - Can a manager see only their department's documents, or all-org documents?
  - Are there any methods that return document data without ownership verification?
- Documents_Handlers.php:
  - Does every handler have `check_admin_referer()` (nonce) at entry?
  - Does every handler have `current_user_can()` (capability) at entry? Or is it nonce-only (same as Hiring Phase 13 HADM-SEC-002)?
  - Are `$_POST`/`$_FILES` inputs sanitized before use?
- Catalogue every `$wpdb` call across all 3 files: line number, method, prepared (yes/no), query type.

**2. Performance:**
- Does the service load all documents for an employee/department without pagination or LIMIT?
- Are there N+1 patterns (e.g., fetching employee details per document in a loop)?
- Does document listing involve information_schema checks (antipattern from Phase 04/08/11/12)?
- Are file operations (disk reads, file_exists checks) done in loops?

**3. Duplication:**
- Is file upload handling duplicated between service and handlers?
- Are there helper functions that duplicate Core/ utilities (e.g., sanitization, capability checks)?
- Compare document upload pattern with Assets module upload (Phase 11) -- is the same insecure pattern repeated?

**4. Logical issues:**
- Document deletion: Does deleting a DB record also delete the physical file? Or do orphaned files accumulate?
- Document versioning: If a document is re-uploaded, is the old file cleaned up?
- Category/type management: Are document categories validated or can arbitrary categories be injected?
- State consistency: Can a document exist in DB but not on disk (or vice versa)?

For each finding, assign severity (Critical/High/Medium/Low) and provide a concrete fix recommendation.
