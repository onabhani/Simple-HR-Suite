<task type="auto">
  <name>Task 1: Read and audit Documents admin tab and REST endpoints</name>
  <files>
    includes/Modules/Documents/Admin/class-documents-tab.php (581 lines -- admin document management tab with views, forms, listing)
    includes/Modules/Documents/Rest/class-documents-rest.php (137 lines -- REST API endpoints for document operations)
  </files>
  <action>
Read both files completely. Audit against the 4 metrics:

**1. Security (Critical focus):**
- class-documents-tab.php:
  - Does the render method have a capability gate at entry (e.g., `current_user_can('sfs_hr.manage')` or similar)?
  - Is document data scoped by department or employee? Or can any admin see all-org documents (same issue as Resignation Phase 14 RES-SEC-001)?
  - Are all echoed values escaped with `esc_html()`, `esc_attr()`, `esc_url()`? Check download URLs, filenames, employee names, dates.
  - Are `$_GET`/`$_POST` params sanitized (`sanitize_text_field`, `absint`, etc.)?
  - If the tab generates download links, can an authenticated user craft a URL to download another employee's document?
  - File download handler: Does it check ownership before serving the file?
- class-documents-rest.php:
  - What permission_callback is used for each route? Flag `__return_true` as Critical (same as Attendance Phase 05).
  - Are REST endpoints properly checking document ownership or just authentication?
  - Is input validation done on REST parameters (document ID, employee_id, file data)?
  - Does the REST layer delegate to Documents_Service or has it inline $wpdb queries?
- Catalogue every `$wpdb` call across both files: line number, method, prepared (yes/no).

**2. Performance:**
- Admin tab: Does the document listing have server-side pagination (SQL LIMIT/OFFSET), or does it load all documents and paginate in PHP?
- Are there N+1 patterns (e.g., fetching employee name per document row in a loop instead of a JOIN)?
- REST: Are list endpoints bounded with per_page/LIMIT, or do they return unbounded result sets?
- Any information_schema or SHOW TABLES calls in these files?

**3. Duplication:**
- Is document listing logic duplicated between admin tab and REST (both querying documents independently)?
- Are there inline SQL queries that should be delegated to Documents_Service?
- Compare admin tab patterns with other admin tabs audited in prior phases (e.g., Assets Phase 11 view files with inline $wpdb).

**4. Logical issues:**
- Document type/category filtering: Does the filter work correctly or can it be bypassed?
- Sorting: Is sort column validated against a whitelist or can arbitrary columns be injected (SQL injection via ORDER BY)?
- Search: Is document search using LIKE with proper escaping (`$wpdb->esc_like()`)?
- Edge cases: What happens when a document's physical file is missing but the DB record exists? Does the admin gracefully handle this?

For each finding, assign severity (Critical/High/Medium/Low) and provide a concrete fix recommendation.
