<objective>
Audit the Leave balance calculation service, leave type configuration, and balance management methods across the Leave module for correctness, security, and performance issues. This plan focuses on the data/calculation layer: tenure-based accrual rules, balance CRUD, leave type definitions, and all $wpdb queries in the services and UI files.

Purpose: Balance calculation correctness is critical — errors here mean employees get wrong leave entitlements. Tenure-based accrual (Saudi labor law: entitlement increases with years of service) has complex edge cases around hire dates, partial periods, and mid-year changes. The LeaveCalculationService.php (243 lines) contains the core formula, while LeaveModule.php (~7390 lines) contains balance management methods, leave type CRUD, and data access that feeds the calculation.
Output: `.planning/phases/06-leave-audit/06-01-leave-services-balance-findings.md`

  <action>
Read all 3 PHP files. For LeaveModule.php, read in chunks (lines 1-2000, 2001-4000, 4001-6000, 6001-7390). Focus this plan's audit on the following concerns:

**1. Balance Calculation Correctness (Critical focus):**
Audit LeaveCalculationService.php thoroughly:
- Tenure-based accrual: Does the formula correctly increase entitlement based on years of service? Saudi labor law grants 21 days for first 5 years, 30 days after. Verify the implementation matches.
- Hire mid-year: If employee started June 15, does pro-rated balance for the first year calculate correctly?
- Partial periods: What happens at year boundaries (e.g., employee crosses 5-year mark mid-year)?
- Leap year handling: Does the calculation account for 365 vs 366 day years?
- Terminated employees: Are balances frozen or still accruing after termination?
- Zero/negative balance: Can balance go negative? Is that intentional?
- Rounding: Are fractional days handled consistently (floor, ceil, round)?

**2. Balance Management in LeaveModule.php:**
Identify and audit all methods related to balance operations:
- get_balance, set_balance, recalculate_balance, adjust_balance (or similar method names)
- Leave type CRUD: create/update/delete leave types -- check for missing capability validation
- Balance initialization: When a new employee is created, how is their balance set?
- Annual reset/carryover: Is there logic for carrying over unused leave to next year?

**3. Security -- $wpdb Query Scan (all 3 files):**
- Every `$wpdb->query()`, `$wpdb->get_results()`, `$wpdb->get_var()`, `$wpdb->get_row()` call -- is the query built with `$wpdb->prepare()` or is user input concatenated/interpolated directly? Flag raw interpolation as Critical.
- Input sanitization: Any `$_POST`, `$_GET`, `$_REQUEST`, `$request->get_param()` used without sanitization in balance or type operations. Flag as High.

**4. Performance:**
- N+1 queries: Balance recalculation that queries per-employee in a loop. Flag as High.
- Unbounded queries: SELECT on leave_balances or leave_types without LIMIT. Flag as Medium.
- Redundant calculations: Same balance computed multiple times without caching. Flag as Medium.

**5. Duplication:**
- Balance calculation logic duplicated between LeaveCalculationService.php and LeaveModule.php. Flag as Medium.
- Date/tenure helper logic repeated instead of shared utility. Flag as Low.

For each finding, record:
1. **Severity**: Critical / High / Medium / Low
2. **Category**: Security, Performance, Duplication, Logic, or Calculation
3. **File and line number(s)**: e.g., `includes/Modules/Leave/Services/LeaveCalculationService.php:L45-L60`
4. **Description**: What the issue is, in one sentence
5. **Evidence**: The problematic code snippet (1-5 lines)
6. **Fix recommendation**: Concrete fix in one sentence

Start with LeaveCalculationService.php (highest business risk -- calculation correctness), then class-leave-ui.php, then balance-related sections of LeaveModule.php. While reading LeaveModule.php, also note the line ranges of approval workflow and overlap detection methods for Plan 02's scope -- but do NOT audit those methods in this plan.
