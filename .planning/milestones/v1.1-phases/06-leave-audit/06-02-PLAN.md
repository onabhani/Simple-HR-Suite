<objective>
Audit the Leave request approval workflow, overlap detection, REST endpoints, and admin AJAX handlers in LeaveModule.php for security, authorization, race conditions, and logical issues. This plan focuses on the workflow and interface layer: how leave requests are created, approved/rejected, how overlapping requests are detected, and how all endpoints are protected.

Purpose: The approval workflow is where authorization bugs are most dangerous — employees approving their own leave, managers acting outside their department, or requests being moved to invalid states. Overlap detection race conditions can allow double-booking of leave. LeaveModule.php (7390 lines) is a monolith containing REST routes, admin AJAX, approval state machine, and overlap checks.

  <action>
Read LeaveModule.php in chunks (lines 1-2000, 2001-4000, 4001-6000, 6001-7390) and class-leave-ui.php. Focus this plan's audit on the following concerns (balance calculation methods were covered in Plan 01 -- skip those here):

**1. Approval Workflow Authorization (Critical focus):**
- State machine: What states can a leave request be in (pending, approved, rejected, cancelled)? Are all transitions validated? Can a request go from rejected back to approved? Flag unauthorized transitions as High.
- Capability checks: Who can approve/reject? Is `sfs_hr.leave.review` checked? Can an employee approve their own request? Can a manager approve requests outside their department? Flag missing checks as Critical.
- Self-service: Can employees only create/cancel their own requests? Is employee_id validated against current user? Flag IDOR as Critical.
- Notification triggers: Are notifications sent on state change? Could a malicious state change trigger misleading notifications?

**2. Overlap Detection Race Conditions (High focus):**
- Overlap check: When an employee submits a leave request, is there a check for existing approved/pending requests that overlap the same dates? Find this logic.
- Race condition: If two requests are submitted simultaneously for overlapping dates, is there a DB-level lock or unique constraint preventing both from being approved? Or is it a read-then-write pattern vulnerable to TOCTOU? Flag as High if no locking.
- Edge cases: Half-day leave overlapping full-day leave. Leave spanning month/year boundary. Leave on public holidays.

**3. REST Endpoint Security:**
- Every `register_rest_route` callback: Does `permission_callback` check appropriate capability? Flag `__return_true` as Critical.
- Input validation: Are request parameters (dates, leave_type_id, employee_id) validated and sanitized?
- Response data: Do list endpoints leak other employees' leave data to unauthorized callers?

**4. Admin AJAX Handler Security:**
- Every `wp_ajax_*` handler: Check for `wp_verify_nonce()` and `current_user_can()`. Flag missing as High.
- Bulk operations: If there are bulk approve/reject handlers, are they individually authorized per request?

**5. $wpdb Query Security (workflow-related queries):**
- Every `$wpdb` call in approval, overlap detection, request CRUD, and admin handler methods. Flag unprepared queries as Critical.

**6. Logical Issues:**
- Leave deduction on approval: Is balance deducted when request is approved? What if approval is reverted -- is balance restored?
- Cancellation after approval: Can an approved leave be cancelled? Is balance restored?
- Date validation: Can a leave request be created for past dates? For dates before hire date?
- Department filtering: Admin list pages -- can HR see all departments? Can managers see only their department?

**7. Output Escaping (class-leave-ui.php + LeaveModule.php admin views):**
- Any HTML output: `esc_html()`, `esc_attr()`, `esc_url()` used? Flag unescaped as Medium.

For each finding, record:
1. **Severity**: Critical / High / Medium / Low
2. **Category**: Security, Performance, Logic, or Workflow
3. **File and line number(s)**
4. **Description**: One sentence
5. **Evidence**: Code snippet (1-5 lines)
6. **Fix recommendation**: Concrete fix

Start with approval workflow methods (highest authorization risk), then overlap detection, then REST endpoints, then admin AJAX handlers, then class-leave-ui.php.
